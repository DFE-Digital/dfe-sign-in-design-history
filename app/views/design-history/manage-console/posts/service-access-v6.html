About service access

  • Changed 'Manage policies to 'Manage service access'

    • A policy is a set of rules that determines who can and cannot access a service. During user testing, we found that ‘Manage policies’ wasn’t descriptive enough and if users were coming in to manage access to a service, the mental link wasn’t there with the terminology ‘Manage policies’.

    We changed 'Manage policies' to 'Manage service access' as an umbrella term above 'policies' so that it would be easier for users to understand.

Create policy

  • Limited the organisation policy

    • We limited the organisation policy to just the organisation and role fields, where users can select multiple fields if necessary.  Users have the ability to filter through to find their organisations, if necessary.

  • Limited the user policy

    • We limited the user policy to just the email and role fields, where users can select multiple fields if necessary.

Conditions

  • Changed 'conditions' to 'rules'

    • Changed ‘conditions’ to ‘rules’ as ‘rules’ is simpler and plainer English, scoring as grade 3 on Hemingway compared to ‘conditions’ which scores as post graduate. ‘Rules’ is also more descriptive as these are business rules the user is adding to the policy, and they encapsulate specific instructions in the technical system.

    While both ‘rules’ and ‘conditions’ can be used to describe criteria or criteria-like elements, the pairing of ‘rules’ with ‘policy’ is more common language usage in computing contexts.

Existing policies

  • Policies already within the system

    • As we have created the MVP, we have not created designs for multiple conditions within a policy (for example, by establishment type, organisation ID or status).

    Therefore, policies within the existing system which have multiple conditions will still be displayed in the MVP, but we are not adding functionality to change or remove these policies. Instead, we have a warning message telling users to create a 'Service Now' request.
